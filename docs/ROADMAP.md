# Trading Bot Roadmap

## 1. Project Goal

이 프로젝트의 최종 목표는 한국투자증권 Open API를 이용하는 국내장 자동매매 봇을 안정적으로 운용하고, 매매 상태와 이벤트를 DB에 축적한 뒤 Vue 기반 웹 대시보드와 분석 기능을 더해 개인용 자동매매 시스템으로 발전시키는 것이다.

현재 전략은 예측형 AI 매매가 아니라 VWAP, 거래량, 가격 흐름을 이용하는 실시간 수급 추종형 단타 전략이다.

```text
상승 흐름이 보이면 진입하고,
흐름이 약해지거나 손절 조건에 도달하면 빠르게 청산한다.
```

장기적으로는 리포트 고도화, 장중 리플레이, 전략 튜닝, 다중 전략, AI·뉴스 기반 스윙 판단까지 확장할 수 있는 구조를 지향한다. 다만 v0.x에서는 기능 확장보다 실전 운용 가능한 최소 구조와 안정성을 우선한다.

## 2. Development Principles

- 현재 시장 범위는 국내장(KRX)으로 한정한다.
- 프로그램은 상시 실행할 수 있지만 신규 진입은 국내장 거래 시간에만 허용한다.
- 신규 진입 허용 시간의 기본값은 `09:10~15:00`으로 유지한다.
- 점심시간을 별도로 차단하지 않고 거래량과 거래대금 조건으로 종목을 필터링한다.
- 예측형 AI보다 VWAP·거래량·가격 흐름 기반의 실시간 수급 추종을 우선한다.
- 작은 기능을 단계적으로 완성하고 실전 운용 결과를 기준으로 개선한다.
- 주문, 체결, 잔고, 전략 판단 데이터를 로그와 DB에 남긴다.
- 추후 리플레이가 가능하도록 이벤트와 전략 판단에 필요한 문맥을 보존한다.
- 대시보드는 복잡한 차트보다 현재 보유 종목과 오늘 매매 이벤트 그리드부터 시작한다.
- 미국장, AI 판단, 자동 튜닝은 국내장 실전 운용이 안정화된 이후 검토한다.

## 3. Current Baseline

현재 저장소에는 다음 기반이 구현되어 있다.

- 한국투자증권 국내장 시세, 주문, 계좌, 잔고, 미체결, 체결 API 연동
- VWAP·거래량 돌파 기반 진입 및 손절·익절·강제 청산 판단
- 거래대금 순위 기반 관심종목 갱신
- 일일 손실, 연속 손실, 동시 보유 수, 중복 주문 제한
- API 재시도, rate limit 대응, safe mode, kill switch
- 텍스트 로그와 구조화 JSONL 이벤트 로그
- SQLite 기반 `orders`, `executions`, `positions`, `account_snapshots` 저장
- 로그와 DB를 이용한 일일 매매 리포트
- `src/broker`, `src/runner`, `src/strategy`, `src/risk`, `src/db`, `src/report` 단위의 기본 코드 분리

따라서 아래 v0.x 단계는 처음부터 새로 만드는 계획이 아니라, 현재 기반을 검증하고 대시보드에서 사용할 계약으로 정리하는 계획이다.

## 4. Version Roadmap

### v0.7 - Stability

목표는 실전 운용 중 오작동과 복구 불가능한 상태를 줄이는 것이다.

- 주문 실패와 불확실한 주문 결과의 재확인 절차 검증
- API 응답 오류와 네트워크 오류 처리 보강
- 시작 시 잔고, 미체결 주문, 당일 체결 복구 검증
- 잔고·보유 종목·체결 동기화의 정합성 강화
- 일일 손실 및 연속 손실 제한 검증
- 신규 진입 차단 조건과 safe mode 전환 기준 정리
- 주문 직후와 주기적 계좌 스냅샷 저장 검증
- 운영 장애 분석에 필요한 로그 필드 보강
- 리플레이에 필요한 전략 설정, 시장 스냅샷, 판단 결과 필드 점검

이 단계에서는 리플레이 기능 자체를 구현하지 않는다.

### v0.8 - Database Logging

목표는 실시간 상태와 이벤트를 DB에 안정적으로 저장하고 대시보드용 최소 조회 데이터를 준비하는 것이다.

상태: 완료

현재 `orders`, `executions`, `positions`, `account_snapshots`, `bot_events` 테이블이 구현되어 있다. JSONL 이벤트 기록과 동일한 구조화 이벤트를 `bot_events`에도 저장하며, DB 저장 실패가 자동매매와 파일 로그 기록을 중단하지 않도록 분리되어 있다.

- 최신 잔고에 없는 종목을 `positions`에서 제거해 현재 보유 상태와 동기화
- 현재 포지션 조회 repository 제공
- 날짜별 주문·체결 조회 repository 제공
- 날짜별 봇 이벤트 조회 repository 제공
- 스킵, 오류, 전략 판단, 주문, 관심종목 이벤트 저장
- 종목명, 이벤트 시각, 이벤트 유형, 매수·매도 방향, 수량, 가격, 사유 저장
- repository 및 runner 동기화 테스트 추가

초기 대시보드에 필요한 최소 데이터는 다음 두 화면을 기준으로 한다.

1. 현재 보유 종목
2. 오늘 매매 및 봇 이벤트

복잡한 통계 테이블과 일간 집계 테이블은 만들지 않는다.

### v0.9 - Project Structure & Dashboard Preparation

목표는 자동매매 프로세스, 대시보드 API, 프론트엔드의 실행 경계를 명확히 하고 웹 대시보드 개발을 준비하는 것이다.

상태: 완료

현재 코드는 이미 `src/` 하위 책임별 패키지로 분리되어 있으므로, 즉시 대규모 이동을 수행하지 않는다. 먼저 배포 단위와 공유 코드 경계를 결정한 뒤 단계적으로 재구성한다.

적용된 구조:

```text
/
├─ apps/
│  ├─ bot/       # 자동매매 프로세스
│  ├─ api/       # 대시보드용 API 서버
│  └─ web/       # Vue 대시보드
├─ src/          # bot runtime 및 공유 DB/config/domain
├─ docs/
├─ reports/
├─ README.md
└─ .gitignore
```

주요 작업:

- 완료: DB, 로그, 리포트, 설정, 토큰 캐시의 runtime path 설정화
- 완료: `apps/api` read-only FastAPI 최소 구조
- 완료: 현재 포지션, 이벤트, 주문, 체결, 계좌 요약 조회 API
- 완료: Dashboard API 계약 문서
- 완료: `apps/web` 프론트엔드 준비 범위 문서
- 완료: SQLite WAL 및 잠금 대기 설정
- 완료: `apps/bot`으로 자동매매 조립 및 CLI 실행 단위 분리
- 완료: `AutoTradingRunner`에서 상태 복구, 잔고·체결 동기화, 계좌 스냅샷 책임 분리
- 완료: 주문 실행, 불확실 주문 재확인, 포지션 갱신 책임 분리
- 완료: bot과 API의 SQLite 동시 읽기·쓰기 자동 테스트
- 완료: `KisOrder`의 broker API 호출과 주문 persistence 분리
- 완료: API 전용 read-only `TradingQueryRepository` 분리
- 완료: broker와 API package의 금지 의존성 자동 테스트
- 완료: bot 전용 코드와 API 공유 코드 경계 확정
- v1.1 예정: 실제 Vue 프로젝트와 Dashboard MVP 구현

프론트엔드 후보:

- Vue 3
- TypeScript
- Vite
- Naive UI 또는 Element Plus

백엔드 후보:

- FastAPI
- 초기 SQLite
- 장기 PostgreSQL 확장 고려

v0.9에서는 대시보드를 완성하지 않는다. API는 로컬 read-only 용도로 제한하며 주문, 봇 제어, 설정 변경 기능을 제공하지 않는다.

### v1.0 - Production Stabilization

상태: 코드 안정화 완료, 장중 검증 진행 필요

- 실제 매매 운용 결과를 기준으로 안정성 검증
- 실패 주문과 미체결 주문 대응 강화
- 일일 손익 리포트 정확성 개선
- 매수될 뻔한 종목과 조건별 탈락 사유 분석
- 운용 당시 전략 파라미터 기록과 추적
- 조건 완화·강화 후보를 근거와 함께 제안
- 장애 복구 절차와 운영 체크리스트 작성

구현 완료:

- 주문 상태를 `OPEN`, `PARTIALLY_FILLED`, `FILLED`로 동기화
- 미체결 제한 시간 초과와 주문 상태 불명 시 Safe Mode
- KIS 실현손익과 내부 계산 손익 차이 저장
- 전략 설정 기반 버전 해시 기록
- 일일 리포트에 손익 정합성과 전략 버전 표시
- read-only `validate` 운영 점검 명령
- 장 시작 전, 장애 대응, 재시작, 장 마감 운영 문서

남은 완료 조건:

- 실제 또는 모의 장중 검증 체크리스트 통과

- 표본 수, 거래비용, 장세 구간을 반영한 결과 해석

- 전략별 주문 및 성과 식별

- 전략 간 동일 종목 주문 충돌 방지
- 단타 전략 외 스윙 전략을 추가할 수 있는 구조
- 사용자 설정 기반 전략 선택

### v3.0 - AI & News Swing Trading

- 실시간 뉴스와 공시 데이터 수집
- 종목별 이슈 분류와 영향 분석
- 며칠간 보유하는 스윙 전략 연구
- AI를 독립 주문 주체가 아닌 운용 보조 판단 계층으로 검토
- 판단 근거, 데이터 출처, 모델 버전 기록
- 오판과 데이터 지연을 고려한 보수적 리스크 제한

이 단계는 국내장 단타 봇과 대시보드가 안정화된 이후의 장기 확장 범위다.

## 8. Milestone Completion Criteria

각 버전은 기능 존재 여부만으로 완료 처리하지 않는다.

- 관련 테스트가 통과한다.
- 실제 또는 dry-run 운용 로그로 정상 흐름과 실패 흐름을 확인한다.
- 운영 중 복구 방법과 제한사항이 문서화되어 있다.
- 민감정보가 로그, DB, 리포트, Git 추적 파일에 노출되지 않는다.
- 다음 단계가 현재 자동매매 프로세스의 안정성을 낮추지 않는다.
